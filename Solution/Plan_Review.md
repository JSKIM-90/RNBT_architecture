# 3트랙 병행 전략 합리성 검토

## 제시된 계획

### Track 1: Figma to Code
- Figma 디자인을 RENOBIT으로 전환하는 프로세스 정립
- 회사 Figma 자산을 템플릿으로 전환 (1회성 퍼블리싱)

### Track 2: fx.js 학습 + 유틸 최적화
- fx.js 깊은 학습
- WKit, GlobalDataPublisher, WEventBus 코드 최적화
- 테스트 코드 작성

### Track 3: Runtime Scaffold 케이스 스터디
- 다양한 시나리오로 패턴 검증
- 유틸의 문제점/부족함 진단
- 패턴 개선

**실행 방식**: 3개 트랙 동시 진행, 비중 조절

---

## 합리성 평가

### ✅ 강점

#### 1. 이론-실습 균형이 좋음
```
Track 2 (학습) → Track 3 (검증) → Track 1 (실전)
└─────────────── 순환 피드백 ──────────────┘

이론만 파지 않음
실습만 하지 않음
→ 균형잡힌 접근
```

#### 2. 실용적 산출물
```
Track 1: 템플릿 자산 (재사용 가능)
Track 2: 최적화된 유틸 (품질 향상)
Track 3: 검증된 패턴 (표준화)

→ 모두 실제 프로젝트에서 사용 가능
```

#### 3. 점진적 개선 구조
```
Scaffold 작성 → 문제 발견 → 유틸 개선 → 다시 Scaffold 작성
└────────────────── 반복 ──────────────────┘

→ 빠른 피드백 루프
```

#### 4. 리스크 분산
```
Track 1만 하면: 퍼블리싱만 잘하게 됨
Track 2만 하면: 이론에 빠짐
Track 3만 하면: 검증 없는 패턴

3개 동시: 어느 하나가 막혀도 다른 트랙으로 전환 가능
```

### ⚠️ 잠재적 문제점

#### 1. 동시 진행의 집중력 분산
```
문제: 3개를 동시에 하면 산만할 수 있음

현실:
- 오전에 fx.js 공부
- 오후에 템플릿 작업
- 저녁에 Scaffold 작성
→ 하루에 3개 컨텍스트 전환

영향: 깊이 있는 학습 방해 가능
```

**대응책**:
```
✓ 주 단위로 비중 조절
  Week 1-2: Track 2 집중 (60%), Track 3 (40%)
  Week 3-4: Track 1 집중 (60%), Track 2 (40%)

✓ 하루 단위 집중
  월/수/금: Track 2 + Track 3
  화/목: Track 1
```

---

#### 2. Track 간 의존성
```
문제:
Track 3 (패턴 검증) → Track 2 (유틸 개선 필요 발견)
→ Track 2 완료 전까지 Track 3 진행 어려움

예시:
Scaffold 작성 중 "pipeForDataMapping에 기능 필요"
→ 유틸 수정
→ 다시 Scaffold 작성
→ 또 문제 발견
→ 무한 루프?
```

**대응책**:
```
✓ 초기에는 유틸 그대로 사용
✓ 문제점만 기록 (즉시 수정 X)
✓ 일정 기간(2주) 후 일괄 개선
✓ "완벽한 유틸" 추구 금지
```

---

#### 3. 범위 확장 유혹
```
위험:
Track 1: "이 템플릿도, 저 템플릿도 변환해야지"
Track 2: "fx.js 말고 함수형 프로그래밍 전체를 공부해야지"
Track 3: "모든 케이스를 다 검증해야지"

→ 끝나지 않음
```

**대응책**:
```
✓ 명확한 범위 설정
  Track 1: 핵심 템플릿 5개만
  Track 2: fx.js 핵심 개념만, 유틸 3개만
  Track 3: 대표 케이스 5개만

✓ 완료 기준 명시
  "완벽"이 아닌 "동작 + 문제점 파악"
```

### 🤔 근본적 질문

#### Q1: 왜 3개를 동시에 해야 하는가?

**A: 서로 피드백을 주기 때문**

```
순차 진행의 문제:
Track 1 완료 → Track 2 시작 → Track 3 시작
└→ 3개월 후 문제 발견: "Track 1에서 만든 템플릿이 패턴과 안 맞음"
   └→ 다시 처음부터

병행 진행:
Track 1 (템플릿 작업) 중 문제 발견
→ Track 3 (패턴)으로 피드백
→ Track 2 (유틸)에서 개선
→ Track 1에 바로 적용

→ 1주 안에 피드백 루프 완성
```

**평가**: ✅ 병행이 합리적

---

#### Q2: JavaScript 작업과 퍼블리싱 작업, 성격이 다른데 병행 가능한가?

**A: 오히려 보완적**

```
JavaScript 작업 (Track 2, 3):
- 논리적 사고
- 추상화
- 깊은 집중 필요

퍼블리싱 작업 (Track 1):
- 시각적 작업
- 구체적
- 가벼운 집중

→ 뇌 사용 영역이 다름
→ 한 쪽에 지치면 다른 쪽으로 전환
→ 번아웃 방지
```

**평가**: ✅ 병행이 오히려 도움됨

---

#### Q3: 혼자서 3개를 동시에 할 수 있는가?

**현실 체크**:
```
일일 작업 시간: 8시간

최소 배분:
Track 1: 2시간 (템플릿 1개 변환)
Track 2: 3시간 (fx.js 학습 + 리팩토링)
Track 3: 2시간 (Scaffold 1개 작성)
여유: 1시간

→ 가능은 함
```

**하지만**:
```
위험:
- 매일 3개 다 하려면 부담
- 품질 저하 가능

현실적 접근:
- 주 3일: Track 2 + Track 3 (JavaScript 집중)
- 주 2일: Track 1 (퍼블리싱 집중)
→ 1주일에 2개 트랙만 활성화
```

**평가**: ⚠️ 동시 진행 맞지만, 일별 집중도 조절 필요
